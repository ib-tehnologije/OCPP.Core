# Project Overview

## Purpose

OCPP.Core provides a .NET-based OCPP central system for managing electric vehicle charge points. The repository includes the OCPP server, an operator management UI, a public charging/payment UI, database model, protocol simulators, and regression tests.

## Current Status

- Repository visibility: public/open-source.
- Primary branch observed locally: `main`.
- CI exists for build/unit tests, browser regression, simulator E2E, and Docker image publishing.
- Current release process: Unknown / verify.
- Current production deployment topology: Unknown / verify.
- Private/internal project context is maintained outside this public repository.

## Main Users and Actors

- Charge points connecting over OCPP 1.6, OCPP 2.0.1, or OCPP 2.1 WebSockets.
- Operators using the management portal to view charge points, connectors, transactions, tags, reports, public portal settings, and remote commands.
- Drivers using the public portal to find/start/monitor paid charging sessions.
- External systems integrated through Stripe, email/SMTP, e-racuni invoice APIs, Azure Service Bus extensions, or custom authorization extensions.
- Maintainers and agents changing server, management, database, integration, or test behavior.

## Major Capabilities

- OCPP WebSocket session handling and protocol message processing for 1.6, 2.0.1, and 2.1.
- Server API for live status and remote actions such as reset, unlock, charging profiles, configuration, start, stop, and payments.
- Operator management portal with authentication, localized UI, charge point and connector views, transaction reports, owner reports, exports, and public portal settings.
- Public charging flow with connector availability, Stripe checkout support, payment reservation lifecycle, status polling, stop requests, and R1 invoice request handling.
- EF Core data model with SQL Server migrations and SQLite local/test support.
- Hosted maintenance services for stale payment reservations, stale connector status release, open transaction recovery, and idle-fee warning emails.
- Optional Hangfire processing for SQL Server-backed jobs and owner report scheduling.
- Runtime extension points for raw OCPP message sinks and external authorization.
- Local simulators and Playwright regression tests.

## Non-Goals Inferred From Current Code

- SQLite is not the migration-backed production path; it is used for local/test runs with `EnsureCreated()`.
- Hangfire-backed recurring jobs are not available without a SQL Server connection string.
- Charger-side reservation profiles are currently disabled in middleware code despite an option type existing.

## Important Domain Concepts

- Charge point: a configured charging station identified by `ChargePointId`.
- Connector: an EVSE connector under a charge point, tracked by `ConnectorStatus`.
- Charge tag: a token/id used for authorization and charging sessions.
- Transaction: the OCPP charging session record, including meter values and stop state.
- Payment reservation: a pre-authorization/checkout lifecycle record that locks a connector until charging starts, times out, completes, or is cancelled.
- Public portal settings: branding, SEO, map, QR, theme, support, and idle-fee display settings for public charging pages.
- Owner report: per-owner transaction and payout reporting generated by the management app.
- Extension: a runtime-loaded DLL implementing raw-message forwarding or external authorization.
